code review

为什么要做 code review？
1. 保证代码质量
2. 发现潜藏bug 
3. 树立团队代码规范
4. 团队之间的职责交替（你了解我的功能，我了解你的功能）


review的形式
1. 一定需要集中在会议室评定吗？ 还是在线上评定？
2. 参与的人员：一定需要一个高级别的程序员参与，或者如果你是修改某块业务逻辑，一定需要这个模块负责人参与。
3. 每次提交的代码量不要太多。否则别人每次review都要话很多时间。
4. 提出来的review意见，其作者一定要及时回复&&订正，否则就失去了意义。
5. review的重点，review的重点是针对本次提交的代码，而不是需求讨论和合理性讨论，千万不要发散思维。





代码分支管理策略




代码规范
需要注意的几点：
1. 一定要是可执行的代码规范。如果代码规范制定了，但是发现推行不下去，那就没有意义了。所以宁可要宽松可执行的制度，也不要严格但不可行的制度。
